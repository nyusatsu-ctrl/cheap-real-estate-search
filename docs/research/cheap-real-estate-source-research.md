
全国の格安不動産収集ソース調査報告
結論
MVPで最初に入れるべきなのは、全国版空き家バンクと県版空き家プラットフォーム、国税庁・裁判所・財務省・KSIの官公庁系売却情報、0円・無償譲渡専門サイト、そして価格判定や危険判定に使う公的データです。理由は、全国または広域で一覧性があり、自治体・官公庁由来の一次情報が多く、0円物件・無償譲渡・低価格物件・農地・山林・公売物件が実際に掲載されており、MVPに必要な「全国から拾えること」「元サイトへ送客できること」「重複・価格変動を追いやすいこと」を満たしやすいからです。LIFULL HOME'S 空き家バンクは全国版空き家バンクの代表で、都道府県ページを持ち、国交省も全国版空き家・空き地バンクとして案内しています。国税庁公売情報は不動産・農地の新着や件数を公開しており、BITは全国ブロック検索・過去データ・スケジュールを備えています。県版では熊本県と栃木県が広域集約型の実例として有力です。

逆に、SUUMO、Yahoo!不動産、アットホーム、楽待、健美家などの大手商用ポータルは情報価値そのものは高いものの、著作権・営業利用・運営妨害に関する条項、403、robots拒否が観測されており、MVPの直接スクレイピング対象としては後回しにするのが安全です。MVPは「一次情報・官公庁・0円/無償譲渡・地方移住系」に寄せ、商用ポータルは提携・広告・送客、またはユーザー検索補助に留める構成が最も現実的です。

自治体空き家バンクの自動発見方法
最も堅い方法は、総務省系の自治体コード/J-LISを自治体マスタにし、国交省の空き家・空き地バンク総合情報ページ、LIFULL HOME'S 空き家バンク、アットホーム空き家バンク、県版プラットフォームをシードURLとして収集し、その後に自治体ドメイン限定検索で不足分を埋める方式です。熊本県や栃木県のような県版集約サイトがある県は、まず県版を取り込み、そこから参加市町村ページを逆引きするのが効率的です。

実装上は、municipalities テーブルに都道府県コード・市区町村コード・名称・旧称・ドメインを持ち、source_discovery_queue に「自治体ごとの検索式」「候補URL」「判定ステータス」を保存すると運用しやすいです。自動発見後は、source_registry に「source_type=municipal_akiya_bank」「discovery_method=google_query|mlit_seed|pref_platform」として登録し、人手レビューを 1 回挟んでから本クロールに移す構成が安全です。

法務・規約・安全な表示設計
民間ポータルの画像転載と本文転載は、原則として避けるのが安全です。LIFULL HOME'S は著作権侵害や営利目的利用、サービス提供妨害行為を禁止しており、アットホームはサイト上の著作物の著作権が自社または権利者に帰属すると明記しています。LINEヤフーの共通利用規約は、コンテンツ利用権が「当社サービスの利用を唯一の目的」とするライセンスであることを示しており、ジモティーの投稿ライセンスもジモティー社への許諾であって、第三者アプリへの自由再配布を意味しません。したがって、私設アグリゲータ側で転載してよいと解釈するのは危険です。

一方で、不動産情報ライブラリAPIは提供情報を無償で利用できる旨が明記され、eMAFF農地ナビやハザードマップポータルも公式利用規約を持っています。このため、MVPでは「私有ポータルの本文・画像は転載しない」「公的データは規約範囲で積極活用する」という切り分けが最も安全です。

安全な表示設計は、次の形がよいです。物件カードでは、価格、都道府県、市区町村、住所一部、土地面積、建物面積、築年数、物件種別、元サイト名、元サイトURL、初回検知日、更新有無、独自スコアだけを表示し、詳細本文・図面・画像は原則表示しません。詳細ページでは、公的補完データによる「ハザード」「用途地域」「農地/山林判定」「路線価近傍」「過去取引価格近傍」などを追加し、ユーザーが最終確認は元サイトで行う導線にします。これは送客型・比較型サービスとして説明可能で、法務リスクを大幅に下げます。

重複判定は必須です。LIFULL HOME'S 空き家バンクやアットホーム空き家バンクは自治体物件を集約しており、自治体個別ページや県版プラットフォームと同じ物件が別URLで掲載される可能性があります。また、自治体が「みんなの0円物件」など外部サイトと連携する例もあります。したがって、source_external_id があれば最優先、なければ「市区町村コード + 価格 + 土地面積 + 建物面積 + 見出し正規化 + 情報提供元」の組み合わせを主キー候補にし、さらに content_hash と類似度スコアで second pass をかけるのがよいです。

地域別検索を安定させるには、都道府県・市区町村・郵便番号・地域ブロックのマスタを別テーブルで管理するのがよいです。市区町村コードは J-LIS と総務省の全国地方公共団体コード、住所補完は日本郵便の郵便番号データ/API を使うと、自治体名変更や住所表記ゆれに耐えやすくなります。地名変更情報を国土地理協会の変更情報で追うと、長期運用でも地域検索が崩れにくくなります。

新着・更新判定仕様
新着 は first_detected_at >= now() - interval '7 day' で判定し、本日追加 は date(first_detected_at) = current_date で十分です。元サイト新着 は source_published_at が取れた場合にのみ別フラグで持ち、source_published_at >= now() - interval '7 day' とします。0円物件 は price_value = 0、300万円以下 は price_value <= 3000000 です。

更新あり は、title_normalized price_value property_type land_area_sqm building_area_sqm address_partial source_published_at source_updated_at の正規化済みセットから content_hash を計算し、前回値と異なるときに true にします。価格だけ変わった場合は changed_fields = ['price']、面積だけ変わった場合は ['land_area_sqm'] のように差分も保存すると、管理画面でのレビューがしやすくなります。

検知時系列 も残すべきです。property_snapshots テーブルを別に持ち、各クロール時点の主キー項目、価格、面積、更新日時、ハッシュを保存しておくと、価格推移・公開終了・再掲載を追跡できます。

クローラー実装方針
Next.js / Node.js / Supabase 前提なら、収集はアプリ本体から分離したジョブ型が運用しやすいです。source_registry にソース定義、crawl_jobs に実行単位、crawl_results に生HTML/抽出JSON、properties に正規化済み物件、property_snapshots に履歴、crawl_errors に失敗ログを持たせる構成がよいです。

アダプタはサイト別に分けます。たとえば adapters/lifullAkiya.ts adapters/ntaKoubai.ts adapters/bit.ts adapters/ksi.ts adapters/zeroEstate.ts adapters/akisol.ts adapters/ieichiba.ts adapters/jmty.ts のようにし、共通I/Fを listPages() -> extractCards() -> normalize() -> diff() で統一します。官公庁系は HTML/PDF 混在が多いので extractDocuments() を別フックにすると拡張しやすいです。

おすすめの実行順は、最初に A ランクだけ実装し、次に B ランク、最後に C/D ランクの検討です。A ランクだけでも十分に差別化できます。順番は、LIFULL 空き家バンク、国税庁公売、BIT、KSI、みんなの0円物件、アキソル、全国0円不動産、家いちば、県版空き家プラットフォーム、山いちば、公的補完データ API の順が現実的です。

取得頻度は、全国版空き家・0円サイト・個人掲示板は 6〜12 時間ごと、官公庁の公売/競売系は 6 時間ごと、国有財産や自治体公有地は 12〜24 時間ごと、補完データは日次〜月次が妥当です。地理/GIS・路線価・郵便番号・行政コードは月次または更新検知時で足ります。

robots.txt と規約チェックはジョブ実行前に必ず行い、source_registry に robots_checked_at terms_checked_at crawl_policy を保存するべきです。crawl_policy は allow allow_with_rate_limit manual_only disallow で十分です。たとえば、健美家は disallow、楽待とアットホーム空き家バンクは manual_only または disallow 寄り、LIFULL 空き家バンクや官公庁ページは allow_with_rate_limit から始めるのが無難です。

管理画面では、新規 更新あり 重複候補 規約要確認 公開終了候補 を整理してレビューできるようにします。特に重複候補は、「自治体ページと全国版が同一物件」「県版サイトと市町村ページが同一物件」「0円サイトと自治体連携情報が同一物件」の3類型で呈示できると実務的です。
